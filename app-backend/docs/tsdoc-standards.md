# TSDoc Standards

Use TSDoc to explain public contracts, non-obvious behavior, and business decisions. Do not use it to restate what a method name already says.

## Where To Use TSDoc

Add TSDoc to:

- Public classes exported by a module.
- Use cases.
- Domain entities and value objects.
- Repository and port interfaces.
- Public methods whose behavior is not obvious.
- DTOs when fields have business meaning or validation rules.

Usually skip TSDoc for:

- Private helper methods.
- Simple constructors.
- Obvious getters and setters.
- NestJS module classes with no behavior.
- Controllers when route decorators and method names are enough.

## Allowed Tags

Use a small tag set:

```text
@remarks
@param
@returns
@throws
@example
```

Avoid broad file headers by default. Prefer documenting exported contracts directly.

## File Headers

Do not add file headers to every file. The professional standard is selective documentation: file headers are useful when the file has a role that is not obvious from its name or exports.

Prefer plain TSDoc file comments or `@packageDocumentation` for package/barrel entry points. Avoid `@fileoverview`; it is common in JSDoc-style codebases but is not the cleanest default for TSDoc.

Use a file-level comment only when the file itself needs context, for example:

- A complex bootstrap file.
- A file that defines a convention used by several modules.
- A compatibility adapter with unusual behavior.
- A public barrel file that explains the module boundary.
- A file with security, logging, or request lifecycle setup.

Example:

```ts
/**
 * Fastify bootstrap for the HTTP API.
 *
 * @remarks
 * Keep framework setup here so feature modules stay focused on application behavior.
 */
```

Avoid file headers like this:

```ts
/**
 * User controller.
 */
```

The file name already says that.

For an `index.ts` or package entry point:

```ts
/**
 * Authentication module public API.
 *
 * @packageDocumentation
 */
```

```ts
 * Synchronizes the local user record with the identity received from Cognito.

  /**
   * Creates the user when it does not exist, otherwise updates profile fields
   * that can change in the identity provider.

   * @returns The local user profile used by the API.
   */
  async execute(input: ISyncAuthenticatedUserInput): Promise<IUserProfileOutput> {
    // ...
  }
}
```

## Port Example

```ts
/**
 * Verifies Cognito tokens and returns the claims required by the application.
 */
export interface ITokenVerifierPort {
  /**
   * Verifies token signature, issuer, audience, and expiration.
   *
   * @param token - Bearer token without the `Bearer` prefix.
   * @returns Verified identity claims.
   * @throws InvalidTokenError when the token cannot be trusted.
   */
  verify(token: string): Promise<IVerifiedIdentityClaims>;
}
```

## DTO Example

```ts
/**
 * User profile returned to authenticated clients.
 */
export class UserProfileResponse {
  /**
   * Public user ID generated by the application database.
   */
  id!: string;

  /**
   * Primary email verified by Cognito.
   */
  email!: string;
}
```

## Comments Inside Functions

Use inline comments only for decisions that are easy to miss:

```ts
// Cognito can change the display name after a Google/Microsoft profile update.
const displayName = claims.name ?? existingUser.displayName;
```

Avoid comments like:

```ts
// Save the user.
await this.userRepository.save(user);
```

## Error Documentation

Use `@throws` only for expected domain/application errors. Do not list every possible infrastructure exception.

Good:

```ts
/**
 * @throws UserNotFoundError when no user exists for the given ID.
 */
```

Avoid:

```ts
/**
 * @throws Error when something fails.
 */
```

## Style Rules

- Keep the first sentence short and specific.
- Use `@remarks` for context, tradeoffs, or business rules.
- Use `@param` only when the parameter needs explanation beyond its name.
- Use `@returns` only when the return value is not obvious.
- Keep examples short and executable in spirit.
- Do not document implementation details that may change often.

## Recommended Enforcement

Use linting for exported APIs once the codebase exists:

```text
eslint-plugin-tsdoc
```

Start with warnings, not hard failures. The goal is useful documentation, not comment volume.
